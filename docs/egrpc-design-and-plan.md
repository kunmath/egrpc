# egrpc — Design & Implementation Plan

**A lightweight, wire-compatible gRPC C++ client for embedded Linux**

Status: Design locked for v0.1 · Target: Yocto-based embedded Linux devices · License: (TBD, suggest Apache-2.0 to match upstream gRPC)

---

## 1. Problem Statement & Goals

Full upstream gRPC C++ is painful on embedded Linux: it drags in Abseil, BoringSSL/OpenSSL, c-ares, re2, zlib, and upb; it requires host-side codegen tools that must not be cross-compiled (often forcing two separate gRPC builds or fragile CMake tool-path overrides); and its binary and build-time footprint is disproportionate for devices that only ever act as clients.

egrpc is a clean-room **client-only** gRPC implementation that is fully wire-compatible with unmodified upstream gRPC servers. Cloud teams keep running standard gRPC stacks; devices link egrpc; both sides share the same `.proto` files with zero server-side changes.

### Design goals

1. **Drop-in stub compatibility — precisely scoped.** Stubs generated by the **pinned `grpc_cpp_plugin` version** (§4.5; v1.82.0 for egrpc v0.1) with **documented plugin options** (the default option set; no `services_namespace`, no mock output) over a **lite-compatible proto graph** — every transitively imported `.proto` must build under `optimize_for = LITE_RUNTIME` — compile and run against egrpc unchanged. We provide the `grpc::internal` API surface that generated code references, with an include tree mirroring `grpcpp/` so generated `#include <grpcpp/...>` lines resolve to our headers. Outside that envelope (other plugin versions, reflection-dependent protos, plugin options we don't test) compatibility is not promised.
2. **Minimal dependency set.** nghttp2 (HTTP/2 framing), OpenSSL (TLS), protobuf-lite (serialization). All three are already common in Yocto BSPs. No Abseil, no c-ares, no re2, no zlib, no upb.
3. **Built for the device traffic pattern.** First-class support for exactly two shapes: device→cloud telemetry over unary RPCs, and cloud→device commands over one long-lived server-streaming RPC (the subscribe-loop pattern, required because devices sit behind NAT/cellular and cannot accept inbound connections).
4. **Resilience as a feature, not an exercise for the reader.** Keepalive PINGs, reconnect with exponential backoff + jitter, deadlines, and clean shutdown are core, tested behaviors — plus a `resilient_stream` helper that packages the subscribe-loop pattern with a reconnect signal.
5. **Trivial to build in Yocto.** A first-party meta-layer with recipes. Codegen tools (`protoc`, `grpc_cpp_plugin`) are pinned host-side tools, never cross-compiled to target.

### Scope

**In scope (v0.1):** unary calls with deadlines; server-streaming (`ClientReader`); TLS via OpenSSL with ALPN negotiating `h2`; HTTP/2 keepalive PING with configurable enforcement; channel reconnect with exponential backoff and jitter; clean shutdown semantics; `resilient_stream` convenience helper; Yocto meta-layer.

**Permanently out of scope:** server-side functionality, client streaming, bidirectional streaming, xDS, interceptors, compression, name resolution beyond DNS (no `c-ares`; use `getaddrinfo`), async completion-queue API.

**Compile-present, runtime-refused:** async stub methods, client-streaming and bidi factories. These exist as symbols so unmodified generated code links, but return/deliver `UNIMPLEMENTED` at runtime. This keeps the "generated code compiles unchanged" promise without implementing machinery devices don't need.

## 2. Architecture Overview

Three layers, strictly separated:

```
┌───────────────────────────────────────────────────────────┐
│  Layer 3: API                                             │
│  include/grpcpp/**    — upstream-compatible shim surface   │
│                          (Channel, ClientContext, Status,  │
│                           ClientReader, grpc::internal::*) │
│  include/egrpc/**     — native conveniences                │
│                          (resilient_stream, config types)  │
├───────────────────────────────────────────────────────────┤
│  Layer 2: Core                                            │
│  ChannelImpl          — connection lifecycle, backoff FSM  │
│  CallState            — per-RPC state machine + buffers    │
│  EventThread          — one per channel; owns everything   │
│                          below this line                   │
├───────────────────────────────────────────────────────────┤
│  Layer 1: Transport                                       │
│  Http2Session         — thin RAII wrapper over nghttp2     │
│  TlsSocket            — OpenSSL BIO, ALPN, non-blocking IO │
│  TimerHeap            — keepalive / backoff / deadlines    │
│  Wakeup (eventfd)     — cross-thread signaling             │
└───────────────────────────────────────────────────────────┘
```

Generated stub code never sees Layers 1–2. It calls `grpc::internal::BlockingUnaryCall(...)` and `grpc::internal::ClientReaderFactory<R>::Create(...)`, which route into `ChannelImpl`.

## 3. Threading Model — Decision

**Decision: (a) one background event thread per channel, with condvar-blocked callers.** The caller-pumped model (b) is rejected.

### Rationale

The deciding constraint is that **nghttp2 sessions are not thread-safe**, and our target workload has two concurrent callers on the same channel by design: a telemetry thread issuing unary calls, and a subscribe-loop thread blocked in `Read()` on the long-lived stream. Under model (b), both would need to pump the same session, forcing a leader-election/handoff scheme that reinvents a dedicated event thread poorly. Three further nails in (b)'s coffin:

- **Keepalive is timer-driven, not caller-driven.** PINGs must fire on schedule even when the application is idle between telemetry sends — that's the whole point of keepalive on NAT/cellular paths (keeping the mapping alive and detecting dead peers). A caller-pumped model only makes progress when someone is calling in.
- **Reconnect must proceed in the background** so that the `resilient_stream` helper can re-establish the subscribe loop without the application driving it.
- **Deadlines need an independent clock.** A caller blocked in a dead `Read()` can't time itself out if it's also the only thing pumping IO.

The cost — one thread per channel — is acceptable: embedded devices in this pattern use exactly one channel. Stack size will be configurable (default 128 KiB via `pthread_attr_setstacksize`) to keep memory bounded.

### Mechanics

The **EventThread** exclusively owns: the socket fd, the `SSL*`, the `nghttp2_session*`, the timer heap, and all `CallState` transitions. Its loop is a `poll()`/`epoll` over `{socket fd, eventfd}` with a timeout equal to the next timer expiry.

Caller threads never touch nghttp2. They interact through two primitives:

1. **Op queue (MPSC, mutex-protected):** callers enqueue ops — `StartCall`, `SendUnary`, `StartStream`, `ReadRequest`, `Cancel`, `Shutdown` — then write to the eventfd to wake the loop.
2. **Per-call completion:** each `CallState` carries a `std::mutex` + `std::condition_variable`. Callers block on a predicate (`unary_done`, `message_available || stream_closed`); the EventThread signals after transitioning state.

This gives the synchronous, blocking API that generated stubs expect, with all HTTP/2 complexity confined to one thread.

## 4. Component Design

### 4.1 `TlsSocket` (Layer 1)

Non-blocking TCP connect, then OpenSSL handshake driven from the event loop (`SSL_ERROR_WANT_READ/WRITE` re-armed via poll). Requirements:

- **Name resolution never blocks the EventThread** (required from M6 on; M3–M5 accept blocking `getaddrinfo` on the EventThread because transport failure is channel-terminal there and nothing else needs the loop). Target design:
  - `TlsSocket::Connect` takes **pre-resolved addresses** (a `sockaddr_storage` list); it owns only connect/handshake.
  - A **resolver helper thread** (lazily started per channel) runs `getaddrinfo` and delivers **generation-tagged** results via one mechanism with two steps: it writes the result into the shared slot (below) under the slot mutex, then — only if the slot is still alive — `Post`s a wakeup op; the EventThread reads the slot and discards results whose generation tag is stale (a reconnect superseded them). A failed `Post` (loop stopping) is benign: the slot flag is already dead or the drain will never need the result. So a slow resolver can never resurrect a dead attempt and never touches channel memory directly.
  - **Address iteration:** attempt addresses in `getaddrinfo` order, one at a time, each bounded by the remaining connect timeout; first success wins; all-failed → `TRANSIENT_FAILURE` with the last error.
  - **Re-resolve on every reconnect attempt** (device networks change; DNS TTLs matter more than resolver latency here). No caching beyond the OS's.
  - **Resolver lifetime:** shutdown must not block on a hung `getaddrinfo`, so the resolver thread must never hold a raw pointer into the channel. It communicates through a small heap-allocated result slot (`shared_ptr`: mutex + alive flag + result) shared with the channel; `Shutdown()` marks the slot dead and **does not join a mid-resolve thread** (it self-detaches and its late result is dropped at the slot). The §5.7 time bound holds even mid-resolve; the only leak-shaped cost is one thread finishing a doomed lookup. **Test-harness note:** that detached thread may still be inside `getaddrinfo` at process exit, which leak/thread checkers in the §7 shutdown tests will flag despite being memory-safe by construction — the harness must either wait for the slot to drain before exiting or carry a documented, narrowly-scoped suppression for exactly this thread (the one sanctioned exception to the no-suppressions rule, because the alternative is an unbounded shutdown).

- **ALPN is mandatory:** offer `h2` only; if the peer doesn't select `h2`, fail the connection with `UNAVAILABLE` and a clear message ("server did not negotiate HTTP/2 via ALPN"). No h2c fallback, no `http/1.1` tolerance.
- Certificate verification on by default (`SSL_VERIFY_PEER` + hostname check via `SSL_set1_host`). Config accepts a root CA bundle path/PEM (device fleets commonly pin a private CA), optional client cert/key for mTLS, and an explicit, loudly-named escape hatch for skipping verification in test rigs only.
- TLS 1.2 minimum.

### 4.2 `Http2Session` (Layer 1)

RAII wrapper around `nghttp2_session_client_new` wiring the standard callbacks (`on_frame_recv`, `on_data_chunk_recv`, `on_stream_close`, `on_header`, `send`/`recv` via memory buffers flushed to `TlsSocket`). Owns:

- **SETTINGS:** advertise `MAX_CONCURRENT_STREAMS` (small, e.g. 8), initial window sizes tuned for constrained memory (see flow control, §5.4). The client-sent setting only limits server-initiated streams (which don't exist in gRPC), so the same value is enforced locally as an admission cap: at most `min(local max_concurrent_streams, server-advertised MAX_CONCURRENT_STREAMS)` calls are active at once; excess calls queue in the channel and submit as slots free.
- **PING transmission and ack tracking** for keepalive (§5.2).
- **GOAWAY handling (v0.1 policy — deliberate):** upstream's "drain old calls on the old connection while new calls use a new one" needs two live sockets, which our one-socket-per-channel design rules out. Instead, on GOAWAY: streams **above** `last_stream_id` (server never processed them) fail with `UNAVAILABLE` and are safely retryable by the application (or `resilient_stream`); streams **at or below** `last_stream_id` are allowed a bounded drain (they may already carry a response) of `goaway_drain_timeout` (own knob, default 5 s — matching the shutdown bound's order of magnitude), after which anything still open fails with `UNAVAILABLE`; then the connection is replaced via the backoff FSM. Caveat for callers: both buckets surface as `UNAVAILABLE`, but only the above-`last_stream_id` bucket is *known*-unprocessed — an at-or-below stream that drained without trailers may have executed server-side, so auto-retrying non-idempotent unary calls on `UNAVAILABLE` is on the application (`resilient_stream`'s server-streaming retries are unaffected — re-subscribing is idempotent by design). This trades a burst of retryable failures for a simpler transport — documented, not accidental.

### 4.3 `CallState` (Layer 2)

One per RPC. A state machine:

```
CREATED → HEADERS_SENT → OPEN → [HALF_CLOSED_LOCAL] → CLOSED
                                        (trailers received / RST / deadline / cancel)
```

Responsibilities:

- **Send path:** serialize the request via protobuf-lite `SerializeToArray`, wrap in the gRPC message frame (§5.1), submit via `nghttp2_submit_request` with a data provider. Unary calls half-close immediately (END_STREAM after the single message).
- **Receive path:** reassemble gRPC messages from DATA chunks. Messages can span multiple HTTP/2 DATA frames and multiple messages can arrive in one frame, so this is a length-prefix scanner over an accumulation buffer with a configurable `max_receive_message_size` (default 4 MiB, tunable down for constrained targets).
- **Trailer capture:** `grpc-status`, `grpc-message` (percent-decoded), custom trailing metadata exposed through `ClientContext`.
- **Trailers-only responses:** a HEADERS frame with END_STREAM carrying `grpc-status` and no DATA is valid and common for immediate errors — handled explicitly.

### 4.4 `ChannelImpl` (Layer 2)

Owns the EventThread and the connectivity FSM:

```
IDLE → CONNECTING → READY → TRANSIENT_FAILURE → CONNECTING → ...
                        ↘ SHUTDOWN
```

Matches gRPC's connectivity semantics closely enough that behavior is unsurprising to gRPC-fluent users. Call admission by state, matching upstream exactly:

| Channel state | Fail-fast call (default) | Wait-for-ready call |
|---|---|---|
| IDLE | queues; triggers connect | queues; triggers connect |
| CONNECTING | queues | queues |
| READY | submits | submits |
| TRANSIENT_FAILURE | fails immediately with `UNAVAILABLE` | queues through the failure/backoff |
| SHUTDOWN | fails with `UNAVAILABLE` (terminal) | fails with `UNAVAILABLE` (terminal) |

The common misreading — "fail-fast fails whenever not READY" — is wrong: fail-fast calls in IDLE/CONNECTING **wait** for the connection attempt to conclude; only an established failure state rejects them. Queued calls of either kind still honor their deadlines (`DEADLINE_EXCEEDED` while queued, M5). `resilient_stream` uses wait-for-ready; plain unary defaults to fail-fast, both matching upstream.

### 4.5 The `grpcpp/` shim surface (Layer 3)

The minimum set of headers/types that `grpc_cpp_plugin`-generated `.grpc.pb.h/.cc` files reference. Inventory (validated against the pinned plugin version's output for `route_guide`):

| Shim type | Notes |
|---|---|
| `grpc::Status`, `grpc::StatusCode` | Full 17-code enum, `error_message()`, `error_details()` |
| `grpc::ClientContext` | deadline, metadata add, `TryCancel()`, trailing metadata access, wait_for_ready |
| `grpc::ChannelInterface`, `grpc::Channel` | opaque handle over `ChannelImpl` |
| `grpc::CreateChannel`, `grpc::CreateCustomChannel` | target parsing: `host:port`, `dns:///host:port` |
| `grpc::SslCredentials`, `grpc::SslCredentialsOptions`, `grpc::ChannelCredentials` | maps onto `TlsSocket` config; `InsecureChannelCredentials` present but returns a channel that fails calls with `UNIMPLEMENTED` in v0.1 (TLS-only), revisit later |
| `grpc::ChannelArguments` | accept-and-ignore unknown args; honor keepalive args (§5.2), max message sizes, user-agent |
| `grpc::ClientReader<R>`, `grpc::ClientReaderInterface<R>` | `Read()`, `Finish()`, `WaitForInitialMetadata()` |
| `grpc::ClientWriter/ClientReaderWriter` + async reader types | compile-present, runtime `UNIMPLEMENTED` |
| `grpc::internal::RpcMethod` | name + method type |
| `grpc::internal::BlockingUnaryCall` / `BlockingUnaryCallImpl` | the unary entry point |
| `grpc::internal::ClientReaderFactory<R>` | the server-streaming entry point |
| `grpc::internal::ClientWriterFactory` / `ClientReaderWriterFactory` / `ClientAsyncResponseReaderHelper` etc. | stubs → `UNIMPLEMENTED` |
| `grpc::protobuf::*` aliases, `grpc::StubOptions` | thin aliases onto protobuf-lite types |

**Critical constraint:** `grpc::internal` is explicitly unstable upstream. Mitigation: **pin one `grpc_cpp_plugin` version per egrpc release** (documented, checked in CI by regenerating stubs and compiling), and structure the shim so supporting adjacent plugin versions is additive. The README states the supported plugin version range explicitly.

Serialization inside the shim goes through a `SerializationTraits`-equivalent specialized for protobuf-lite `MessageLite`, so we never need full protobuf reflection. Shared proto files add `option optimize_for = LITE_RUNTIME` — wire-neutral, ignored by Python/Go/Java servers.

### 4.6 `egrpc::resilient_stream<R>` (Layer 3, native API)

The productized subscribe-loop. Wraps `stub->Subscribe(...)`-style server-streaming calls with:

- Automatic re-establishment on stream/channel failure, driven by the channel's backoff FSM.
- A **reconnected signal**: after each successful re-establishment, the helper invokes a user callback (or the first `Read()` after reconnect returns a distinguishable event) so the device can reconcile state it may have missed — resync desired config, re-report status, etc. This is the piece naive retry loops always get wrong.
- Clean stop: `stop()` cancels the in-flight RPC, prevents re-establishment, and unblocks any blocked `Read()`.

**Contract (binding on M5/M6 design, stated now because it constrains their completion signaling):**

- **Retryable vs terminal statuses — classified by origin where the code is ambiguous.** Retry (with backoff) on `UNAVAILABLE`, `RESOURCE_EXHAUSTED`, `ABORTED`, `INTERNAL`, `UNKNOWN`, and **server-originated `CANCELLED`** (RST_STREAM CANCEL from the peer — a bounced stream is transport-shaped); terminal on **locally-originated `CANCELLED`** (`stop()`/`TryCancel` — the helper knows which it is; never classify by status code alone), `UNIMPLEMENTED`, `UNAUTHENTICATED`, `PERMISSION_DENIED`, `INVALID_ARGUMENT`, `FAILED_PRECONDITION`, `NOT_FOUND`, `OUT_OF_RANGE`, `ALREADY_EXISTS`, `DATA_LOSS`. `INTERNAL`/`UNKNOWN` are deliberately retryable: on this transport they are overwhelmingly middlebox/wire-shaped, and the §5.3 READY-stability backoff prevents hammering a genuinely broken server. The set is configurable.
- **Deadlines are per-attempt.** A deadline set in the factory's `ClientContext` applies to each establishment attempt individually and `DEADLINE_EXCEEDED` is retryable — a fresh attempt re-arms it. `resilient_stream` deliberately has no end-to-end deadline (it is an indefinite subscribe loop; ending it is `stop()`'s job). Documented loudly: a deadline here means "give up on *this attempt* after T," never "stop the loop after T."
- **First establishment does NOT fire `on_reconnected`.** The callback signals *re*-establishment — "you may have missed something." The initial connect missed nothing; devices that need connect-time reconciliation do it before entering the read loop. (Configurable via `fire_on_first_connect` for apps that want uniform handling.)
- **Re-establishment is a new RPC, driven by the helper — Layer 2 never resurrects a stream.** Each attempt = tear down the failed `ClientReader`, invoke the user's factory lambda again (fresh `ClientContext`, wait-for-ready), producing a new call with its own completion block (§5.7). There is no Layer-2 auto-restart hook to find. "Successfully re-established" — the `on_reconnected` trigger — is defined as **initial metadata received on the new attempt** (`WaitForInitialMetadata()` returning true); a stream that dies immediately after still counts as a re-establishment (the callback may fire again after the next attempt — "at least once per successful establishment," exactly-once only per §7's stable-network test conditions).
- **Callback threading: never the EventThread.** `on_reconnected` (and any future callbacks) run on the caller's thread inside `Read()` — i.e. between the reconnect completing and the next message being returned. Layer 2 needs no reconnect-detection API for this: the helper itself drove the new attempt from inside `Read()`, so it knows. Nothing in Layer 2 may call user code. A callback that blocks delays only its own stream's reads, never the channel's IO.
- **Reentrancy & exceptions.** Callbacks must not call back into the same `resilient_stream` (`Read`/`stop` from inside `on_reconnected` is undefined; documented, asserted in debug). An exception thrown from a callback propagates out of `Read()` to the caller — the helper does not swallow it; the stream remains valid.
- **`stop()` guarantee:** after `stop()` returns, no callback is running or will run, and no re-establishment will occur. `stop()` may block briefly waiting for an in-flight callback on another thread to return (it never deadlocks when called *from* the callback thread itself — that case is the documented-undefined reentrancy above). Idempotent; safe from any thread except inside a callback.
- **Terminal-status retrieval:** after `Read()` returns `false`, `status()` returns the terminal `grpc::Status` (why the loop ended: the terminal status, or `CANCELLED` after `stop()`). Calling `status()` before `Read()` returned false is valid and returns OK-so-far.
- **Explicitly the application's job:** resume tokens/cursors (telling the server where to resume), replay of missed messages, and deduplication of re-delivered ones. The helper guarantees *at-least-once stream establishment with a reconcile signal*, nothing about message-level continuity.

Sketch of the call-site we're designing for:

```cpp
egrpc::ResilientStreamOptions opts;
opts.on_reconnected = [&]{ reconcile_state(); };

egrpc::resilient_stream<Command> cmds(opts, [&](grpc::ClientContext& ctx) {
    return stub->SubscribeCommands(&ctx, MakeSubscribeRequest());
});

Command cmd;
while (cmds.Read(&cmd)) {   // blocks; survives disconnects transparently
    handle(cmd);
}
```

And the telemetry side stays plain upstream-style gRPC:

```cpp
grpc::ClientContext ctx;
ctx.set_deadline(std::chrono::system_clock::now() + 5s);
grpc::Status s = stub->ReportTelemetry(&ctx, reading, &ack);
```

## 5. Protocol Details (the correctness-critical fine print)

### 5.1 gRPC over HTTP/2 wire format

Request HEADERS: `:method: POST`, `:scheme: https`, `:authority: <host[:port]>`, `:path: /pkg.Service/Method`, `te: trailers`, `content-type: application/grpc`, `user-agent: grpc-c++-egrpc/<ver>`, `grpc-timeout: <N><unit>` when a deadline is set (unit chosen for max precision that fits 8 digits: n/u/m/S/M/H), plus `ClientContext` metadata (lowercase keys; `-bin` suffixed keys base64-encoded without padding).

Each message on the wire: `[1-byte compressed flag = 0][4-byte big-endian length][payload]`. We send `grpc-encoding: identity` semantics implicitly and set `grpc-accept-encoding: identity`; a server responding with compressed messages (flag=1) fails the call with `INTERNAL` ("compression not supported") — servers won't compress unless the client advertises support, so this is a defensive check.

Response handling: initial HEADERS must be `:status: 200` with `content-type: application/grpc*`; then DATA; then trailers with `grpc-status` (int) and optional `grpc-message` (percent-encoded UTF-8). Trailers-only responses handled per §4.3. Missing `grpc-status` at a clean stream end falls back to the HTTP-status mapping (§5.5), so a 200 without `grpc-status` → `UNKNOWN`, matching upstream; a malformed `grpc-status` value likewise maps to `UNKNOWN`.

### 5.2 Keepalive

Implements gRPC's client keepalive semantics on top of nghttp2 PING:

- `keepalive_time` (default **300 s** — server-safe: matches upstream servers' default `min_ping_interval_without_data`, so an unmodified server never GOAWAYs on first contact; configurable via `GRPC_ARG_KEEPALIVE_TIME_MS` through `ChannelArguments` for shim parity or `egrpc::ChannelConfig` natively): send PING after this long with no reads.
- **Cellular profile** (opt-in, not default): NAT-mapping keepalive typically wants `keepalive_time = 60 s`. This trips a default upstream server's `too_many_pings` enforcement, so deployments choosing it **must** coordinate server settings (`GRPC_ARG_HTTP2_MIN_RECV_PING_INTERVAL_WITHOUT_DATA_MS ≤ 60 s` and `GRPC_ARG_KEEPALIVE_PERMIT_WITHOUT_CALLS = 1` on the server). M7's `egrpc::ChannelConfig` ships this as a named preset with these requirements in its documentation. Against a non-coordinated server the ENHANCE_YOUR_CALM doubling below makes the client degrade rather than break.
- `keepalive_timeout` (default 20 s): if no PING ack (or any read) within this window, declare the connection dead → `TRANSIENT_FAILURE`, fail in-flight calls with `UNAVAILABLE`, enter backoff.
- `keepalive_permit_without_calls` (default **true** for egrpc — deviating deliberately from upstream's default, because the NAT-mapping use case is the whole point; documented).
- Respect server `GOAWAY` with `ENHANCE_YOUR_CALM`/`too_many_pings` debug data by doubling `keepalive_time` and logging loudly — this is the standard defense against tripping server-side keepalive enforcement.

### 5.3 Reconnect backoff

gRPC's standard algorithm: `initial_backoff = 1 s`, `multiplier = 1.6`, `max_backoff = 120 s`, `jitter = ±20 %` (uniform). All parameters configurable; cellular deployments may want a higher floor.

Two deliberate deltas from the published algorithm (grpc/doc/connection-backoff.md):

- **`MIN_CONNECT_TIMEOUT = 20 s`, unified with `connect_timeout`.** The published algorithm gives each attempt `max(MIN_CONNECT_TIMEOUT, current_backoff)` to complete. We already default `connect_timeout` to 20 s; that value *is* our `MIN_CONNECT_TIMEOUT` — one knob, published-algorithm semantics: an attempt gets `max(connect_timeout, current_backoff)` before being declared failed.
- **READY-stability reset instead of reset-on-connect.** The published algorithm resets backoff to initial on any successful connection. A server that accepts connections and immediately falls over (crash loop, overloaded LB) would then see us hammering at `initial_backoff` forever. We reset only after a connection has held READY for a stability threshold (default: 60 s, configurable); a connection that dies sooner continues the previous backoff progression. Deliberate anti-flap variant, documented as a deviation.

### 5.4 Flow control (memory-bounding on constrained devices)

Two hard constraints shape this design. First, the receive window must always admit a full message **plus its framing**: a window of exactly `max_receive_message_size` deadlocks on a max-size message because the 5-byte gRPC frame prefix doesn't fit (a 4 MiB window cannot carry 4 MiB + 5 bytes). Second, nghttp2's automatic window management (`nghttp2_option_set_no_auto_window_update`) is **session-wide** — it cannot be automatic for unary streams and manual for the long-lived stream, so the earlier "manual for streams, automatic for unary" phrasing is unimplementable as written.

The scheme:

- **Auto window updates disabled globally** at session creation. All credit is returned explicitly via `nghttp2_session_consume()` (and its `_stream`/connection split where needed).
- **Connection-level credit is returned immediately** on every DATA chunk, for all streams — connection window must never be the throttle, or one slow stream would starve the whole channel (including keepalive-adjacent traffic and other calls).
- **Unary streams return stream credit immediately** as well: their memory is already bounded by `max_receive_message_size` via the reassembly scanner, and the response is consumed as one unit.
- **Long-lived (server-streaming) stream credit is deferred to `Read()`**, and the hop is explicit because §3 forbids callers touching nghttp2: `Read()` pops a decoded message locally under the completion-block mutex (empty queue → block on the condvar), then `Post`s a `ReadRequest` op; **on the EventThread** that op calls `nghttp2_session_consume_stream()` for the drained bytes and flushes the resulting WINDOW_UPDATE. This caller-pop/EventThread-credit split is the most race-prone piece of M5 — the completion-block mutex guards the queue, the op queue carries the credit, nothing else crosses the boundary. A fast server pushing at a slow device exhausts the stream window and blocks **on the wire** instead of ballooning device memory.
- **Window sizing — derived, not independent:** `stream_window = max_receive_message_size + frame prefix (5 B) + slack` (slack default 64 KiB: HTTP/2 frame headers, several in-flight messages' prefixes, and room so the window never reaches exactly zero at a message boundary). The window is **computed from the message cap at channel creation** — it must not be an independently-settable knob, or a user setting both to the same value reintroduces the deadlock this section exists to prevent; an explicit window override, if ever exposed, is validated `≥ max_receive_message_size + 5`. (The current `Http2SessionOptions::initial_window_size` field becomes derived in M5.) The advertised `SETTINGS_INITIAL_WINDOW_SIZE` and the connection window both use the inflated value; default ≈ 4.07 MiB for the 4 MiB default message cap.
- **Delivery-queue bounds:** deferring credit bounds *wire* intake, but decoded messages waiting in the per-stream delivery queue are also bounded: at most `max_queued_messages` (default 16) **or** `max_queued_bytes` (default = stream window) — whichever trips first stops the EventThread from *decoding* further. Undelivered bytes then sit in the **reassembly buffer** (they must be copied out of nghttp2 in `on_data_chunk_recv` — nghttp2 reclaims the chunk when the callback returns, so nothing can be "left inside" nghttp2), with their stream credit withheld, so the wire stalls. Total per-stream memory ≤ window + queue bound, deterministically.

### 5.5 Error mapping

When trailers carry `grpc-status`, it wins. Otherwise map per the gRPC HTTP/2 spec:

| Condition | Status |
|---|---|
| HTTP `:status` 400 | INTERNAL |
| 401 | UNAUTHENTICATED |
| 403 | PERMISSION_DENIED |
| 404 | UNIMPLEMENTED |
| 429 / 502 / 503 / 504 | UNAVAILABLE |
| RST_STREAM REFUSED_STREAM | UNAVAILABLE (safe to transparently retry — noted for future) |
| RST_STREAM CANCEL | CANCELLED |
| RST_STREAM ENHANCE_YOUR_CALM | RESOURCE_EXHAUSTED |
| RST_STREAM INADEQUATE_SECURITY | PERMISSION_DENIED |
| RST_STREAM other codes | INTERNAL (matches upstream `grpc_http2_error_to_grpc_status`) |
| Connection lost / GOAWAY mid-call | UNAVAILABLE |
| Local deadline expiry | DEADLINE_EXCEEDED (and send RST_STREAM CANCEL) |
| `TryCancel()` | CANCELLED |
| Malformed gRPC frame (bad flag, oversize, truncated) | INTERNAL |
| Missing grpc-status on clean 200 close, or malformed grpc-status value | UNKNOWN (upstream falls back to HTTP-status mapping; 200 → UNKNOWN) |

### 5.6 Deadlines & cancellation

Deadlines on unary calls are v0.1 scope: `ClientContext::set_deadline` → `grpc-timeout` header **and** a local timer in the EventThread's timer heap; whichever fires first wins locally. Expiry: RST_STREAM(CANCEL) on the wire, `DEADLINE_EXCEEDED` to the caller. `TryCancel()` enqueues a `Cancel` op — safe from any thread, idempotent. Server-streaming calls take no deadline by default (they're intentionally long-lived) but honor one if set.

### 5.7 Shutdown semantics

Channel destruction (or explicit `Shutdown()`): (1) refuse new ops with `UNAVAILABLE`, (2) `TryCancel` all in-flight calls, (3) unblock all condvar waiters, (4) send GOAWAY-adjacent graceful close (`nghttp2_session_terminate_session(NO_ERROR)`), flush, `SSL_shutdown` best-effort with a short bounded timeout, close, (5) join the EventThread. **Guarantee:** channel destruction never hangs longer than the configured shutdown timeout (default 5 s) and never crashes even with live `ClientReader`s — readers' subsequent `Read()` returns false and `Finish()` returns `UNAVAILABLE`. `resilient_stream::stop()` layers on top of this.

**Ownership graph (who may outlive whom, and what they hold):**

- The **EventThread owns the protocol side of every CallState** — transport buffers, stream bookkeeping, transitions. Protocol-side state dies with the channel; nothing outside the EventThread ever touches it.
- The **reader/caller holds a `shared_ptr` to a completion block** — a small heap object containing exactly: the per-call `std::mutex` + `std::condition_variable`, the terminal `Status`, received metadata slots, and the delivery queue (streaming). The EventThread holds a second reference for as long as the call is live and drops it at terminal transition. Because the completion block references **no transport object**, a `ClientReader` (or a blocked `Wait()`) may outlive `ChannelImpl` destruction safely: shutdown writes the terminal result into the block, signals the condvar, drops its reference — the caller's reference keeps the block alive arbitrarily long after the channel is gone. (M3/M4's `CallState` already has the load-bearing *lifetime* property — `shared_ptr`, no transport references — but not the object split: today protocol state and completion state live in one object, and the EventThread holds it in two places (the `active_calls_` map and the captured `StreamHooks`). The M5 split moves both of those to the protocol-side object; it is a real refactor, not a rename.)
- **`TryCancel()` is a sticky token, valid before submission — and it is both mechanisms of §5.6, not either.** It (a) sets a flag in the completion block from any thread at any point in the call's life — including after the op is queued but before the EventThread admits it — **and** (b) enqueues a `Cancel` op to wake the loop. The flag alone would leave a mid-flight cancel waiting for unrelated IO; the op alone would lose the pre-admission race. The EventThread checks the token at admission and at each transition; a pre-submission cancel means the call terminates `CANCELLED` without ever touching the wire. Cancelling a completed call is a no-op. Never blocks.
- **Shutdown is an admission gate under the op-queue mutex.** `Shutdown()` closes the gate first (new ops are refused with `UNAVAILABLE` at enqueue, under the same mutex that orders the queue), *then* drains: every op that made it through the gate reaches **exactly one** terminal result — completed, failed, or cancelled — before the EventThread exits. No op is silently dropped; no waiter is left blocked. This pair (gate + drain-to-terminal) is the invariant that makes the 5 s bound provable rather than probabilistic.

### 5.8 `ClientReader` lifecycle contract

Legal call sequences and their semantics (upstream-compatible where upstream defines behavior; explicit where upstream is silent):

- **`Read(msg)`** — blocks for the next message. Returns `true` with a message; `false` at end-of-stream (trailers received, stream reset, cancel, deadline, or channel shutdown — cause reported by `Finish()`). After the first `false`, every later `Read()` returns `false` immediately. Single-reader: concurrent `Read()`s on one reader are undefined (documented; asserted in debug).
- **`Finish()`** — returns the terminal `Status`; blocks until the stream reaches a terminal state. **Calling `Finish()` without having drained the stream cancels-and-discards**: it sends RST_STREAM(CANCEL), discards queued messages, and waits only for the reset to confirm (no credit bookkeeping is needed at this point — connection credit was returned immediately per §5.4, and the reset closes the stream window with it). This is mandatory, not a convenience — with Read-driven credit (§5.4) an undrained stream holds the window closed; a `Finish()` that politely waited for trailers behind megabytes of undelivered messages the app has promised never to read would block on a server that is itself blocked on our window: a self-inflicted deadlock. Upstream requires draining before `Finish()`; we accept the call and make it mean "I'm done — cancel the rest." At most once; second call undefined (assert in debug).
- **`WaitForInitialMetadata()`** — blocks until initial metadata arrives (or terminal state). Optional; legal only before the first `Read()`; idempotent after arrival.
- **Destruction** — destroying a reader without `Finish()` performs the cancel-and-discard of `Finish()` and does not block on the network (the completion block absorbs the terminal result nobody will read). Destruction during a concurrent `Read()`/`Finish()` on another thread is undefined, as upstream.
- **Ordering:** `WaitForInitialMetadata()`* → `Read()`* → `Finish()` → destruction is the canonical sequence; any prefix is legal. `Finish()` before any `Read()` is legal (= immediate cancel unless trailers already arrived). Nothing is legal after `Finish()` except destruction.

## 7. Validation Strategy

Three concentric rings, all against **unmodified upstream servers**:

**Ring 1 — C++ unit tests** (run everywhere, no network): gRPC message-frame scanner (split frames, coalesced messages, oversize rejection), `grpc-timeout` encoding, percent-decoding of `grpc-message`, base64 `-bin` metadata, backoff sequence with jitter bounds, timer heap ordering, status mapping table.

**Ring 2 — route_guide acceptance** (the compatibility oracle): upstream gRPC C++ `route_guide` example with only its includes swapped to egrpc; protoc/`grpc_cpp_plugin` output **unchanged**; `option optimize_for = LITE_RUNTIME` added to the proto (wire-neutral). In scope: `GetFeature` (unary) and `ListFeatures` (server-streaming); `RecordRoute`/`RouteChat` stubbed and asserted to return `UNIMPLEMENTED`. Counterpart: Python gRPC server with TLS.

**Ring 3 — behavior matrix** (pytest harness owning server lifecycle, shelling out to built example/test binaries):

- TLS: successful ALPN h2; server without ALPN → clean `UNAVAILABLE`; bad cert → `UNAVAILABLE` with handshake detail; mTLS happy path.
- Keepalive: idle connection stays alive past NAT-ish timeouts (assert PINGs on the wire via server-side debug hooks or a packet-counting TCP proxy in the harness); server that blackholes traffic (harness proxy drops packets) → client detects death within `keepalive_timeout` and reconnects.
- Reconnect: kill/restart server mid-stream → `resilient_stream` re-establishes, `on_reconnected` fires exactly once per re-establishment, backoff timing within jitter bounds (harness measures connection-attempt timestamps).
- Deadlines: slow server → `DEADLINE_EXCEEDED` locally *and* server sees cancellation.
- Errors/trailers: server aborts with each status code + message + custom trailing metadata → verify propagation byte-for-byte.
- Shutdown: destroy channel with live stream → bounded time, reader unblocks correctly; run under ASan/TSan in CI.
- **Interop-suite subset** (credibility mechanism): run applicable cases from the upstream gRPC interoperability test suite — `empty_unary`, `large_unary`, `server_streaming`, `status_code_and_message`, `custom_metadata`, `timeout_on_sleeping_server`, `cancel_after_begin`, `unimplemented_method` — against the upstream interop server. Client-streaming/bidi/compression cases excluded by design and documented as such.

A small TCP fault-injection proxy (Python, part of the harness) provides packet blackholing, latency injection, and mid-stream connection resets without needing root or tc/netem.

## 9. Risks & Mitigations

**`grpc::internal` API drift** is the top risk — upstream may change generated-code contracts between plugin versions. Mitigation: pin per release, CI-diff regenerated stubs (M4), keep the shim entry points thin so adaptation is localized, document the supported range. **nghttp2 callback subtleties** (partial writes, deferred data providers) are the second: mitigated by M2/M3 being pure-transport milestones with fault-injection tests before any shim work. **Keepalive vs server enforcement**: defaults chosen conservatively and ENHANCE_YOUR_CALM backoff implemented from day one. **TSan/ASan in CI from M1 onward** is non-negotiable given the condvar-heavy design — data races here would be brutal to debug on-device later.
