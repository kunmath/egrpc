# egrpc — Design & Implementation Plan

**A lightweight, wire-compatible gRPC C++ client for embedded Linux**

Status: Design locked for v0.1 · Target: Yocto-based embedded Linux devices · License: (TBD, suggest Apache-2.0 to match upstream gRPC)

---

## 1. Problem Statement & Goals

Full upstream gRPC C++ is painful on embedded Linux: it drags in Abseil, BoringSSL/OpenSSL, c-ares, re2, zlib, and upb; it requires host-side codegen tools that must not be cross-compiled (often forcing two separate gRPC builds or fragile CMake tool-path overrides); and its binary and build-time footprint is disproportionate for devices that only ever act as clients.

egrpc is a clean-room **client-only** gRPC implementation that is fully wire-compatible with unmodified upstream gRPC servers. Cloud teams keep running standard gRPC stacks; devices link egrpc; both sides share the same `.proto` files with zero server-side changes.

### Design goals

1. **Drop-in stub compatibility.** Code generated by upstream `protoc` + `grpc_cpp_plugin` compiles and runs against egrpc unchanged. We provide the `grpc::internal` API surface the generated code references, with an include tree mirroring `grpcpp/` so generated `#include <grpcpp/...>` lines resolve to our headers.
2. **Minimal dependency set.** nghttp2 (HTTP/2 framing), OpenSSL (TLS), protobuf-lite (serialization). All three are already common in Yocto BSPs. No Abseil, no c-ares, no re2, no zlib, no upb.
3. **Built for the device traffic pattern.** First-class support for exactly two shapes: device→cloud telemetry over unary RPCs, and cloud→device commands over one long-lived server-streaming RPC (the subscribe-loop pattern, required because devices sit behind NAT/cellular and cannot accept inbound connections).
4. **Resilience as a feature, not an exercise for the reader.** Keepalive PINGs, reconnect with exponential backoff + jitter, deadlines, and clean shutdown are core, tested behaviors — plus a `resilient_stream` helper that packages the subscribe-loop pattern with a reconnect signal.
5. **Trivial to build in Yocto.** A first-party meta-layer with recipes. Codegen tools (`protoc`, `grpc_cpp_plugin`) are pinned host-side tools, never cross-compiled to target.

### Scope

**In scope (v0.1):** unary calls with deadlines; server-streaming (`ClientReader`); TLS via OpenSSL with ALPN negotiating `h2`; HTTP/2 keepalive PING with configurable enforcement; channel reconnect with exponential backoff and jitter; clean shutdown semantics; `resilient_stream` convenience helper; Yocto meta-layer.

**Permanently out of scope:** server-side functionality, client streaming, bidirectional streaming, xDS, interceptors, compression, name resolution beyond DNS (no `c-ares`; use `getaddrinfo`), async completion-queue API.

**Compile-present, runtime-refused:** async stub methods, client-streaming and bidi factories. These exist as symbols so unmodified generated code links, but return/deliver `UNIMPLEMENTED` at runtime. This keeps the "generated code compiles unchanged" promise without implementing machinery devices don't need.

## 2. Architecture Overview

Three layers, strictly separated:

```
┌───────────────────────────────────────────────────────────┐
│  Layer 3: API                                             │
│  include/grpcpp/**    — upstream-compatible shim surface   │
│                          (Channel, ClientContext, Status,  │
│                           ClientReader, grpc::internal::*) │
│  include/egrpc/**     — native conveniences                │
│                          (resilient_stream, config types)  │
├───────────────────────────────────────────────────────────┤
│  Layer 2: Core                                            │
│  ChannelImpl          — connection lifecycle, backoff FSM  │
│  CallState            — per-RPC state machine + buffers    │
│  EventThread          — one per channel; owns everything   │
│                          below this line                   │
├───────────────────────────────────────────────────────────┤
│  Layer 1: Transport                                       │
│  Http2Session         — thin RAII wrapper over nghttp2     │
│  TlsSocket            — OpenSSL BIO, ALPN, non-blocking IO │
│  TimerHeap            — keepalive / backoff / deadlines    │
│  Wakeup (eventfd)     — cross-thread signaling             │
└───────────────────────────────────────────────────────────┘
```

Generated stub code never sees Layers 1–2. It calls `grpc::internal::BlockingUnaryCall(...)` and `grpc::internal::ClientReaderFactory<R>::Create(...)`, which route into `ChannelImpl`.

## 3. Threading Model — Decision

**Decision: (a) one background event thread per channel, with condvar-blocked callers.** The caller-pumped model (b) is rejected.

### Rationale

The deciding constraint is that **nghttp2 sessions are not thread-safe**, and our target workload has two concurrent callers on the same channel by design: a telemetry thread issuing unary calls, and a subscribe-loop thread blocked in `Read()` on the long-lived stream. Under model (b), both would need to pump the same session, forcing a leader-election/handoff scheme that reinvents a dedicated event thread poorly. Three further nails in (b)'s coffin:

- **Keepalive is timer-driven, not caller-driven.** PINGs must fire on schedule even when the application is idle between telemetry sends — that's the whole point of keepalive on NAT/cellular paths (keeping the mapping alive and detecting dead peers). A caller-pumped model only makes progress when someone is calling in.
- **Reconnect must proceed in the background** so that the `resilient_stream` helper can re-establish the subscribe loop without the application driving it.
- **Deadlines need an independent clock.** A caller blocked in a dead `Read()` can't time itself out if it's also the only thing pumping IO.

The cost — one thread per channel — is acceptable: embedded devices in this pattern use exactly one channel. Stack size will be configurable (default 128 KiB via `pthread_attr_setstacksize`) to keep memory bounded.

### Mechanics

The **EventThread** exclusively owns: the socket fd, the `SSL*`, the `nghttp2_session*`, the timer heap, and all `CallState` transitions. Its loop is a `poll()`/`epoll` over `{socket fd, eventfd}` with a timeout equal to the next timer expiry.

Caller threads never touch nghttp2. They interact through two primitives:

1. **Op queue (MPSC, mutex-protected):** callers enqueue ops — `StartCall`, `SendUnary`, `StartStream`, `ReadRequest`, `Cancel`, `Shutdown` — then write to the eventfd to wake the loop.
2. **Per-call completion:** each `CallState` carries a `std::mutex` + `std::condition_variable`. Callers block on a predicate (`unary_done`, `message_available || stream_closed`); the EventThread signals after transitioning state.

This gives the synchronous, blocking API that generated stubs expect, with all HTTP/2 complexity confined to one thread.

## 4. Component Design

### 4.1 `TlsSocket` (Layer 1)

Non-blocking TCP connect via `getaddrinfo` (honoring a connect timeout), then OpenSSL handshake driven from the event loop (`SSL_ERROR_WANT_READ/WRITE` re-armed via poll). Requirements:

- **ALPN is mandatory:** offer `h2` only; if the peer doesn't select `h2`, fail the connection with `UNAVAILABLE` and a clear message ("server did not negotiate HTTP/2 via ALPN"). No h2c fallback, no `http/1.1` tolerance.
- Certificate verification on by default (`SSL_VERIFY_PEER` + hostname check via `SSL_set1_host`). Config accepts a root CA bundle path/PEM (device fleets commonly pin a private CA), optional client cert/key for mTLS, and an explicit, loudly-named escape hatch for skipping verification in test rigs only.
- TLS 1.2 minimum.

### 4.2 `Http2Session` (Layer 1)

RAII wrapper around `nghttp2_session_client_new` wiring the standard callbacks (`on_frame_recv`, `on_data_chunk_recv`, `on_stream_close`, `on_header`, `send`/`recv` via memory buffers flushed to `TlsSocket`). Owns:

- **SETTINGS:** advertise `MAX_CONCURRENT_STREAMS` (small, e.g. 8), initial window sizes tuned for constrained memory (see flow control, §5.4).
- **PING transmission and ack tracking** for keepalive (§5.2).
- **GOAWAY handling:** mark channel as draining; in-flight calls run to completion where possible; new calls trigger reconnect.

### 4.3 `CallState` (Layer 2)

One per RPC. A state machine:

```
CREATED → HEADERS_SENT → OPEN → [HALF_CLOSED_LOCAL] → CLOSED
                                        (trailers received / RST / deadline / cancel)
```

Responsibilities:

- **Send path:** serialize the request via protobuf-lite `SerializeToArray`, wrap in the gRPC message frame (§5.1), submit via `nghttp2_submit_request` with a data provider. Unary calls half-close immediately (END_STREAM after the single message).
- **Receive path:** reassemble gRPC messages from DATA chunks. Messages can span multiple HTTP/2 DATA frames and multiple messages can arrive in one frame, so this is a length-prefix scanner over an accumulation buffer with a configurable `max_receive_message_size` (default 4 MiB, tunable down for constrained targets).
- **Trailer capture:** `grpc-status`, `grpc-message` (percent-decoded), custom trailing metadata exposed through `ClientContext`.
- **Trailers-only responses:** a HEADERS frame with END_STREAM carrying `grpc-status` and no DATA is valid and common for immediate errors — handled explicitly.

### 4.4 `ChannelImpl` (Layer 2)

Owns the EventThread and the connectivity FSM:

```
IDLE → CONNECTING → READY → TRANSIENT_FAILURE → CONNECTING → ...
                        ↘ SHUTDOWN
```

Matches gRPC's connectivity semantics closely enough that behavior is unsurprising to gRPC-fluent users. Calls issued while not READY either wait for readiness (wait-for-ready semantics, default for the stream helper) or fail fast with `UNAVAILABLE` (default for plain unary, matching upstream).

### 4.5 The `grpcpp/` shim surface (Layer 3)

The minimum set of headers/types that `grpc_cpp_plugin`-generated `.grpc.pb.h/.cc` files reference. Inventory (validated against the pinned plugin version's output for `route_guide`):

| Shim type | Notes |
|---|---|
| `grpc::Status`, `grpc::StatusCode` | Full 17-code enum, `error_message()`, `error_details()` |
| `grpc::ClientContext` | deadline, metadata add, `TryCancel()`, trailing metadata access, wait_for_ready |
| `grpc::ChannelInterface`, `grpc::Channel` | opaque handle over `ChannelImpl` |
| `grpc::CreateChannel`, `grpc::CreateCustomChannel` | target parsing: `host:port`, `dns:///host:port` |
| `grpc::SslCredentials`, `grpc::SslCredentialsOptions`, `grpc::ChannelCredentials` | maps onto `TlsSocket` config; `InsecureChannelCredentials` present but returns a channel that fails calls with `UNIMPLEMENTED` in v0.1 (TLS-only), revisit later |
| `grpc::ChannelArguments` | accept-and-ignore unknown args; honor keepalive args (§5.2), max message sizes, user-agent |
| `grpc::ClientReader<R>`, `grpc::ClientReaderInterface<R>` | `Read()`, `Finish()`, `WaitForInitialMetadata()` |
| `grpc::ClientWriter/ClientReaderWriter` + async reader types | compile-present, runtime `UNIMPLEMENTED` |
| `grpc::internal::RpcMethod` | name + method type |
| `grpc::internal::BlockingUnaryCall` / `BlockingUnaryCallImpl` | the unary entry point |
| `grpc::internal::ClientReaderFactory<R>` | the server-streaming entry point |
| `grpc::internal::ClientWriterFactory` / `ClientReaderWriterFactory` / `ClientAsyncResponseReaderHelper` etc. | stubs → `UNIMPLEMENTED` |
| `grpc::protobuf::*` aliases, `grpc::StubOptions` | thin aliases onto protobuf-lite types |

**Critical constraint:** `grpc::internal` is explicitly unstable upstream. Mitigation: **pin one `grpc_cpp_plugin` version per egrpc release** (documented, checked in CI by regenerating stubs and compiling), and structure the shim so supporting adjacent plugin versions is additive. The README states the supported plugin version range explicitly.

Serialization inside the shim goes through a `SerializationTraits`-equivalent specialized for protobuf-lite `MessageLite`, so we never need full protobuf reflection. Shared proto files add `option optimize_for = LITE_RUNTIME` — wire-neutral, ignored by Python/Go/Java servers.

### 4.6 `egrpc::resilient_stream<R>` (Layer 3, native API)

The productized subscribe-loop. Wraps `stub->Subscribe(...)`-style server-streaming calls with:

- Automatic re-establishment on stream/channel failure, driven by the channel's backoff FSM.
- A **reconnected signal**: after each successful re-establishment, the helper invokes a user callback (or the first `Read()` after reconnect returns a distinguishable event) so the device can reconcile state it may have missed — resync desired config, re-report status, etc. This is the piece naive retry loops always get wrong.
- Clean stop: `stop()` cancels the in-flight RPC, prevents re-establishment, and unblocks any blocked `Read()`.

Sketch of the call-site we're designing for:

```cpp
egrpc::ResilientStreamOptions opts;
opts.on_reconnected = [&]{ reconcile_state(); };

egrpc::resilient_stream<Command> cmds(opts, [&](grpc::ClientContext& ctx) {
    return stub->SubscribeCommands(&ctx, MakeSubscribeRequest());
});

Command cmd;
while (cmds.Read(&cmd)) {   // blocks; survives disconnects transparently
    handle(cmd);
}
```

And the telemetry side stays plain upstream-style gRPC:

```cpp
grpc::ClientContext ctx;
ctx.set_deadline(std::chrono::system_clock::now() + 5s);
grpc::Status s = stub->ReportTelemetry(&ctx, reading, &ack);
```

## 5. Protocol Details (the correctness-critical fine print)

### 5.1 gRPC over HTTP/2 wire format

Request HEADERS: `:method: POST`, `:scheme: https`, `:authority: <host[:port]>`, `:path: /pkg.Service/Method`, `te: trailers`, `content-type: application/grpc`, `user-agent: grpc-c++-egrpc/<ver>`, `grpc-timeout: <N><unit>` when a deadline is set (unit chosen for max precision that fits 8 digits: n/u/m/S/M/H), plus `ClientContext` metadata (lowercase keys; `-bin` suffixed keys base64-encoded without padding).

Each message on the wire: `[1-byte compressed flag = 0][4-byte big-endian length][payload]`. We send `grpc-encoding: identity` semantics implicitly and set `grpc-accept-encoding: identity`; a server responding with compressed messages (flag=1) fails the call with `INTERNAL` ("compression not supported") — servers won't compress unless the client advertises support, so this is a defensive check.

Response handling: initial HEADERS must be `:status: 200` with `content-type: application/grpc*`; then DATA; then trailers with `grpc-status` (int) and optional `grpc-message` (percent-encoded UTF-8). Trailers-only responses handled per §4.3. Missing `grpc-status` at a clean stream end falls back to the HTTP-status mapping (§5.5), so a 200 without `grpc-status` → `UNKNOWN`, matching upstream; a malformed `grpc-status` value likewise maps to `UNKNOWN`.

### 5.2 Keepalive

Implements gRPC's client keepalive semantics on top of nghttp2 PING:

- `keepalive_time` (default **300 s** — server-safe: matches upstream servers' default `min_ping_interval_without_data`, so an unmodified server never GOAWAYs on first contact; configurable via `GRPC_ARG_KEEPALIVE_TIME_MS` through `ChannelArguments` for shim parity or `egrpc::ChannelConfig` natively): send PING after this long with no reads.
- **Cellular profile** (opt-in, not default): NAT-mapping keepalive typically wants `keepalive_time = 60 s`. This trips a default upstream server's `too_many_pings` enforcement, so deployments choosing it **must** coordinate server settings (`GRPC_ARG_HTTP2_MIN_RECV_PING_INTERVAL_WITHOUT_DATA_MS ≤ 60 s` and `GRPC_ARG_KEEPALIVE_PERMIT_WITHOUT_CALLS = 1` on the server). M7's `egrpc::ChannelConfig` ships this as a named preset with these requirements in its documentation. Against a non-coordinated server the ENHANCE_YOUR_CALM doubling below makes the client degrade rather than break.
- `keepalive_timeout` (default 20 s): if no PING ack (or any read) within this window, declare the connection dead → `TRANSIENT_FAILURE`, fail in-flight calls with `UNAVAILABLE`, enter backoff.
- `keepalive_permit_without_calls` (default **true** for egrpc — deviating deliberately from upstream's default, because the NAT-mapping use case is the whole point; documented).
- Respect server `GOAWAY` with `ENHANCE_YOUR_CALM`/`too_many_pings` debug data by doubling `keepalive_time` and logging loudly — this is the standard defense against tripping server-side keepalive enforcement.

### 5.3 Reconnect backoff

gRPC's standard algorithm: `initial_backoff = 1 s`, `multiplier = 1.6`, `max_backoff = 120 s`, `jitter = ±20 %` (uniform). Backoff resets after a connection has been READY for longer than a threshold (avoid reset-on-flap). All parameters configurable; cellular deployments may want a higher floor.

### 5.4 Flow control (memory-bounding on constrained devices)

Stream-level: initial window sized to `min(4 MiB, max_receive_message_size + slack)`, but **WINDOW_UPDATE for the long-lived stream is tied to application consumption** — the EventThread only releases window as `Read()` drains the reassembly buffer. This makes a fast server pushing commands at a slow device exert backpressure on the wire instead of ballooning device memory. Unary calls use automatic window updates (bounded by max message size anyway).

### 5.5 Error mapping

When trailers carry `grpc-status`, it wins. Otherwise map per the gRPC HTTP/2 spec:

| Condition | Status |
|---|---|
| HTTP `:status` 400 | INTERNAL |
| 401 | UNAUTHENTICATED |
| 403 | PERMISSION_DENIED |
| 404 | UNIMPLEMENTED |
| 429 / 502 / 503 / 504 | UNAVAILABLE |
| RST_STREAM REFUSED_STREAM | UNAVAILABLE (safe to transparently retry — noted for future) |
| RST_STREAM CANCEL | CANCELLED |
| RST_STREAM ENHANCE_YOUR_CALM | RESOURCE_EXHAUSTED |
| RST_STREAM INADEQUATE_SECURITY | PERMISSION_DENIED |
| RST_STREAM other codes | INTERNAL (matches upstream `grpc_http2_error_to_grpc_status`) |
| Connection lost / GOAWAY mid-call | UNAVAILABLE |
| Local deadline expiry | DEADLINE_EXCEEDED (and send RST_STREAM CANCEL) |
| `TryCancel()` | CANCELLED |
| Malformed gRPC frame (bad flag, oversize, truncated) | INTERNAL |
| Missing grpc-status on clean 200 close, or malformed grpc-status value | UNKNOWN (upstream falls back to HTTP-status mapping; 200 → UNKNOWN) |

### 5.6 Deadlines & cancellation

Deadlines on unary calls are v0.1 scope: `ClientContext::set_deadline` → `grpc-timeout` header **and** a local timer in the EventThread's timer heap; whichever fires first wins locally. Expiry: RST_STREAM(CANCEL) on the wire, `DEADLINE_EXCEEDED` to the caller. `TryCancel()` enqueues a `Cancel` op — safe from any thread, idempotent. Server-streaming calls take no deadline by default (they're intentionally long-lived) but honor one if set.

### 5.7 Shutdown semantics

Channel destruction (or explicit `Shutdown()`): (1) refuse new ops with `UNAVAILABLE`, (2) `TryCancel` all in-flight calls, (3) unblock all condvar waiters, (4) send GOAWAY-adjacent graceful close (`nghttp2_session_terminate_session(NO_ERROR)`), flush, `SSL_shutdown` best-effort with a short bounded timeout, close, (5) join the EventThread. **Guarantee:** channel destruction never hangs longer than the configured shutdown timeout (default 5 s) and never crashes even with live `ClientReader`s — readers' subsequent `Read()` returns false and `Finish()` returns `UNAVAILABLE`. `resilient_stream::stop()` layers on top of this.

## 7. Validation Strategy

Three concentric rings, all against **unmodified upstream servers**:

**Ring 1 — C++ unit tests** (run everywhere, no network): gRPC message-frame scanner (split frames, coalesced messages, oversize rejection), `grpc-timeout` encoding, percent-decoding of `grpc-message`, base64 `-bin` metadata, backoff sequence with jitter bounds, timer heap ordering, status mapping table.

**Ring 2 — route_guide acceptance** (the compatibility oracle): upstream gRPC C++ `route_guide` example with only its includes swapped to egrpc; protoc/`grpc_cpp_plugin` output **unchanged**; `option optimize_for = LITE_RUNTIME` added to the proto (wire-neutral). In scope: `GetFeature` (unary) and `ListFeatures` (server-streaming); `RecordRoute`/`RouteChat` stubbed and asserted to return `UNIMPLEMENTED`. Counterpart: Python gRPC server with TLS.

**Ring 3 — behavior matrix** (pytest harness owning server lifecycle, shelling out to built example/test binaries):

- TLS: successful ALPN h2; server without ALPN → clean `UNAVAILABLE`; bad cert → `UNAVAILABLE` with handshake detail; mTLS happy path.
- Keepalive: idle connection stays alive past NAT-ish timeouts (assert PINGs on the wire via server-side debug hooks or a packet-counting TCP proxy in the harness); server that blackholes traffic (harness proxy drops packets) → client detects death within `keepalive_timeout` and reconnects.
- Reconnect: kill/restart server mid-stream → `resilient_stream` re-establishes, `on_reconnected` fires exactly once per re-establishment, backoff timing within jitter bounds (harness measures connection-attempt timestamps).
- Deadlines: slow server → `DEADLINE_EXCEEDED` locally *and* server sees cancellation.
- Errors/trailers: server aborts with each status code + message + custom trailing metadata → verify propagation byte-for-byte.
- Shutdown: destroy channel with live stream → bounded time, reader unblocks correctly; run under ASan/TSan in CI.
- **Interop-suite subset** (credibility mechanism): run applicable cases from the upstream gRPC interoperability test suite — `empty_unary`, `large_unary`, `server_streaming`, `status_code_and_message`, `custom_metadata`, `timeout_on_sleeping_server`, `cancel_after_begin`, `unimplemented_method` — against the upstream interop server. Client-streaming/bidi/compression cases excluded by design and documented as such.

A small TCP fault-injection proxy (Python, part of the harness) provides packet blackholing, latency injection, and mid-stream connection resets without needing root or tc/netem.

## 8. Implementation Plan (milestones sized for Claude Code sessions)

Each milestone is independently landable with green CI, and each has a crisp acceptance test so a Claude Code session has an unambiguous "done." Recommended repo bootstrap: check this document in as `docs/DESIGN.md` and distill §§3–5 plus the milestone list into `CLAUDE.md` so every session has the invariants (thread-ownership rules, wire format, pinned plugin policy) in context.

**M0 — Scaffold.** Repo layout, CMake skeleton building an empty lib against system nghttp2/OpenSSL/protobuf-lite, `tools/pin-codegen.sh` fetching pinned protoc + `grpc_cpp_plugin` with checksum verification, GitHub Actions running unit-test skeleton + clang-format/clang-tidy. *Accept:* `cmake && make && ctest` green in CI on x86-64; codegen tool pinning script produces working binaries.

**M1 — Transport bring-up.** `TlsSocket` (non-blocking connect, OpenSSL handshake, ALPN h2-only, cert verification), `Wakeup` (eventfd), `TimerHeap`, `EventThread` skeleton (poll loop, op queue, timer dispatch). *Accept:* integration test connects to a Python TLS server, asserts `h2` negotiated, asserts clean failure against a server offering only `http/1.1`; unit tests for timer heap + wakeup under TSan.

**M2 — HTTP/2 session + keepalive plumbing.** `Http2Session` wrapper, SETTINGS exchange, PING send/ack tracking wired to the timer heap, GOAWAY handling, connection preface over the M1 socket. *Accept:* against a real HTTP/2 server (the Python gRPC server counts), session establishes, exchanges SETTINGS, PINGs on schedule; blackhole test detects dead peer within `keepalive_timeout`.

**M3 — Hand-rolled unary call, end-to-end.** gRPC framing (send/reassembly scanner), `CallState`, headers construction, trailer parsing, status mapping, condvar completion — driven by a temporary internal API, no generated stubs yet. *Accept:* raw-API test performs `route_guide.GetFeature` against the Python server and round-trips a protobuf-lite message; trailers-only error path verified; framing scanner unit tests (split/coalesced/oversize) green.

**M4 — Shim: unary via unmodified generated stubs.** `grpcpp/` header tree, `Status`, `ClientContext` (metadata + deadline field), `Channel`/`CreateChannel`/`SslCredentials`, `internal::RpcMethod`, `internal::BlockingUnaryCall`, serialization traits for `MessageLite`. Vendor route_guide example with includes swapped. *Accept:* upstream-generated `route_guide.grpc.pb.cc` compiles unchanged against egrpc; example binary's `GetFeature` path passes against the Python server; CI regenerates stubs from the pinned plugin and diffs against checked-in copies.

**M5 — Server-streaming + deadlines + cancellation.** `internal::ClientReaderFactory`, `ClientReader::Read/Finish/WaitForInitialMetadata`, flow-control-tied-to-Read backpressure, `grpc-timeout` + local deadline timers, `TryCancel`. Stub out writer/bidi/async factories → `UNIMPLEMENTED`. *Accept:* `ListFeatures` streams correctly; slow-server deadline test; cancel-mid-stream test; `RecordRoute`/`RouteChat` return `UNIMPLEMENTED`; backpressure test (slow reader bounds memory, measured via RSS ceiling in harness).

**M6 — Resilience: reconnect, keepalive enforcement, shutdown.** Connectivity FSM, backoff with jitter, wait-for-ready, ENHANCE_YOUR_CALM handling, full shutdown sequence per §5.7. *Accept:* server kill/restart tests, backoff-timing assertions, blackhole → reconnect, shutdown-with-live-reader under ASan/TSan, channel destruction bounded-time guarantee.

**M7 — `resilient_stream` + native config API.** The helper per §4.6, `egrpc::ChannelConfig`, `on_reconnected` semantics, `stop()`. *Accept:* subscribe-loop demo survives 20 scripted server restarts with exactly-once `on_reconnected` per re-establishment and zero missed/duplicated notifications; documented reconcile-on-reconnect example.

**M8 — Interop subset, Yocto layer, docs.** Interop client binary implementing the Ring-3 subset; `meta-egrpc` with library recipe, `-native` codegen recipe demonstrating the host-tool pattern, and an example image `.bbappend`; README with supported-plugin-version policy, size/footprint measurements vs full gRPC (the headline number for the project), porting guide. *Accept:* interop subset green against upstream interop server; `bitbake egrpc` succeeds in a reference Yocto build (kirkstone or scarthgap LTS); documented stripped `.so`/static-link size comparison.

Suggested post-v0.1 backlog (explicitly not now): retry policy for `REFUSED_STREAM`, insecure channels for on-box testing, additional plugin-version support, connection-level flow-control tuning knobs, `epoll`-free `poll()`-only build for exotic targets.

---

## 9. Risks & Mitigations

**`grpc::internal` API drift** is the top risk — upstream may change generated-code contracts between plugin versions. Mitigation: pin per release, CI-diff regenerated stubs (M4), keep the shim entry points thin so adaptation is localized, document the supported range. **nghttp2 callback subtleties** (partial writes, deferred data providers) are the second: mitigated by M2/M3 being pure-transport milestones with fault-injection tests before any shim work. **Keepalive vs server enforcement**: defaults chosen conservatively and ENHANCE_YOUR_CALM backoff implemented from day one. **TSan/ASan in CI from M1 onward** is non-negotiable given the condvar-heavy design — data races here would be brutal to debug on-device later.
