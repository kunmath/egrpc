# Pre-M5 Yocto feasibility spike (design §8a)

**Status: procedure ready, not yet executed — needs a Yocto build host**
(~80 GB disk, ~1–3 h first build). The analytic half of the gate is done:
scarthgap ships protobuf 4.25.3 (meta-oe), whose C++ runtime hard-rejects
pre-25.0 gencode (`PROTOBUF_MIN_PROTOC_VERSION 4025000`), which drove the
per-runtime message-codegen matrix in docs/codegen-pinning.md. This spike
confirms the matrix empirically. It is throwaway — the polished layer is M8.

## What green means

`bitbake egrpc-spike` completes on scarthgap: the M4 artifact (libegrpc +
the route_guide unary example) cross-compiles against scarthgap's
nghttp2/OpenSSL/protobuf-lite, with message gencode regenerated in-recipe by
`protobuf-native` (4.25.x) and stubs regenerated by the pinned
`grpc_cpp_plugin` (or checked-in stub copies for the spike — the stub
contract is runtime-independent).

## Procedure

```sh
# 1. Build host setup (Ubuntu 22.04/24.04)
sudo apt install gawk wget git diffstat unzip texinfo gcc build-essential \
     chrpath socat cpio python3 python3-pip xz-utils zstd liblz4-tool file

# 2. Environment + layers
source poky/oe-init-build-env build
bitbake-layers add-layer ../meta-openembedded/meta-oe
bitbake-layers add-layer <path-to>/egrpc/meta-egrpc-spike   # layer below

# 3. The gate
bitbake egrpc-spike
```

## Spike layer content (`meta-egrpc-spike/`, create at repo root when running)

`conf/layer.conf`:

```
BBPATH .= ":${LAYERDIR}"
BBFILES += "${LAYERDIR}/recipes-spike/*/*.bb"
BBFILE_COLLECTIONS += "egrpc-spike"
BBFILE_PATTERN_egrpc-spike = "^${LAYERDIR}/"
LAYERSERIES_COMPAT_egrpc-spike = "scarthgap"
```

`recipes-spike/egrpc/egrpc-spike_0.1.bb` (adjust `SRC_URI`/`SRCREV` to the
commit under test; `file://` + `externalsrc` is fine for a local tree):

```
SUMMARY = "egrpc pre-M5 feasibility spike (design §8a) — throwaway"
LICENSE = "CLOSED"

SRC_URI = "git://github.com/<you>/egrpc.git;protocol=https;branch=main"
SRCREV = "${AUTOREV}"
S = "${WORKDIR}/git"

DEPENDS = "nghttp2 openssl protobuf protobuf-native"

inherit cmake pkgconfig

# The checked-in message gencode targets CI's protobuf 3.21.12 and must not
# be compiled here (docs/codegen-pinning.md): regenerate against the target
# runtime with the lockstep protobuf-native protoc.
do_configure:prepend() {
    rm -f ${S}/examples/route_guide/gen/route_guide.pb.*
    protoc --proto_path=${S}/examples/route_guide \
           --cpp_out=${S}/examples/route_guide/gen \
           ${S}/examples/route_guide/route_guide.proto
}

EXTRA_OECMAKE = "-DBUILD_TESTING=OFF"
```

## What to record on completion

In docs/codegen-pinning.md, replace "validated analytically … confirms it
empirically" with the concrete result: scarthgap release used, recipe
versions actually resolved (protobuf, nghttp2, openssl), and any deviation
needed from the recipe above. If the build FAILS on the protobuf matrix, the
failure mode and version pins go there too — that is the point of gating
before M5, while the codegen policy is still cheap to change.

## Known-unknowns this spike settles

1. protobuf-lite 4.25.x accepts in-recipe 25.x gencode for a
   `LITE_RUNTIME` proto graph (expected: yes, exact-minor match).
2. egrpc's CMake finds scarthgap's nghttp2/OpenSSL via pkg-config in a
   cross environment (expected: yes; `pkgconfig` inherit).
3. The stub gencode (`.grpc.pb.*`, plugin v1.82.0) compiles against
   protobuf 4.25.x headers — the shim's `grpc::protobuf` aliases touch only
   the stable `MessageLite` surface, but 4.x header churn is exactly the
   kind of thing analysis misses.
4. C++17 + `-Wall -Wextra -Werror` under scarthgap's default GCC 13
   cross-toolchain.
