# egrpc — project guide for Claude Code

Authoritative design: **docs/egrpc-design-and-plan.md**. This file is a distillation;
when in doubt, the design doc wins. Do not deviate from decisions marked resolved
there (threading model, flow control, keepalive defaults, TLS-only v0.1) without
asking the user first.

## Purpose & scope

egrpc is a clean-room, **client-only** gRPC C++ library for embedded Linux (Yocto),
wire-compatible with unmodified upstream gRPC servers: code generated by upstream
`protoc` + `grpc_cpp_plugin` compiles and runs against egrpc unchanged. Deps are
nghttp2 + OpenSSL + protobuf-lite only. In scope (v0.1): unary calls with deadlines,
server-streaming `ClientReader`, TLS with ALPN `h2`, keepalive PING, reconnect with
backoff+jitter, clean shutdown, `egrpc::resilient_stream`. Permanently out of scope:
server side, client/bidi streaming, xDS, interceptors, compression, c-ares, async
completion queues. Async/client-streaming/bidi symbols are compile-present but
return `UNIMPLEMENTED` at runtime.

## Threading-ownership invariants (design §3) — never violate

- One background **EventThread per channel**. It **exclusively owns**: the socket
  fd, the `SSL*`, the `nghttp2_session*`, the timer heap, and **all CallState
  transitions**. No other thread may touch these, ever.
- Caller threads interact through exactly two primitives:
  1. a mutex-protected MPSC **op queue** (`StartCall`, `SendUnary`, `StartStream`,
     `ReadRequest`, `Cancel`, `Shutdown`) + eventfd wakeup;
  2. per-call `std::mutex` + `std::condition_variable` completion — callers block
     on a predicate, the EventThread signals after transitioning state.
- Event loop = `poll()`/`epoll` over {socket fd, eventfd}, timeout = next timer
  expiry. Keepalive, reconnect, and deadlines are EventThread-timer-driven and must
  make progress with no application calls in flight.

## gRPC wire format essentials (design §5.1)

- Request HEADERS: `:method: POST`, `:scheme: https`, `:authority: host[:port]`,
  `:path: /pkg.Service/Method`, `te: trailers`, `content-type: application/grpc`,
  `user-agent: grpc-c++-egrpc/<ver>`, `grpc-timeout: <N><unit>` when deadline set
  (n/u/m/S/M/H, max precision fitting 8 digits). Metadata keys lowercase; `-bin`
  keys base64-encoded **without padding**.
- Every message: `[1-byte compressed flag = 0][4-byte big-endian length][payload]`.
  We advertise `grpc-accept-encoding: identity`; a received flag=1 fails the call
  with `INTERNAL`. Messages can span DATA frames and coalesce within one — the
  receive path is a length-prefix scanner over an accumulation buffer bounded by
  `max_receive_message_size` (default 4 MiB).
- Response: initial HEADERS `:status: 200` + `content-type: application/grpc*`,
  then DATA, then trailers with `grpc-status` (int) and optional `grpc-message`
  (percent-encoded UTF-8). **Trailers-only** responses (HEADERS with END_STREAM
  carrying `grpc-status`, no DATA) are valid and must be handled. Missing or
  malformed `grpc-status` at a clean stream end → `UNKNOWN` (HTTP-status
  fallback; 200 → UNKNOWN). When trailers carry `grpc-status` it wins over
  HTTP-status mapping (table in design §5.5).

## Pinned codegen tools (design §4.5)

`grpc::internal` is unstable upstream, so **one `grpc_cpp_plugin` version is pinned
per egrpc release**, verified in CI by regenerating stubs and compiling.

- Pinned for v0.1: **gRPC v1.82.0** (`grpc_cpp_plugin`, built from SHA-256-verified
  source — upstream publishes no prebuilt plugin binaries) and **protoc 35.0**
  (official prebuilt binary, SHA-256-verified), the protobuf version gRPC v1.82.0
  pins. `tools/pin-codegen.sh` fetches/builds both into `tools/bin/`; versions and
  checksums are recorded in that script and docs/codegen-pinning.md.
- Never regenerate stubs with an unpinned tool. Shim changes for a new plugin
  version must be additive.

## Milestones (design §8)

- **M0 — Scaffold (CMake, pin-codegen.sh, CI, test skeleton): done**
- **M1 — Transport bring-up (TlsSocket, Wakeup, TimerHeap, EventThread skeleton): done**
- **M2 — HTTP/2 session + keepalive plumbing: done**
- **M3 — Hand-rolled unary call end-to-end: done**
- **M4 — Shim: unary via unmodified generated stubs: done**
- M5 — Server-streaming + deadlines + cancellation: not started
- M6 — Resilience: reconnect, keepalive enforcement, shutdown: not started
- M7 — resilient_stream + native config API: not started
- M8 — Interop subset, Yocto layer, docs: not started

Each milestone lands independently with green CI and its acceptance test from §8.

## Delegation rules

- KEEP in the main session: all architecture and design-interpretation decisions,
  the EventThread/CallState/ChannelImpl core logic, anything touching the
  threading-ownership invariants, and final integration of subagent output.
- DELEGATE to subagents: unit tests, shim header boilerplate, CMake/CI config, the
  Python test server and pytest harness, shell tooling, docs, and mechanical
  refactors — always with a precise spec, never "figure out X".
- Every delegated task must state its acceptance check (a command that must pass);
  verify subagent output in the main session before committing. Run code-reviewer
  on the diff before each commit.

Subagents (in `.claude/agents/`): `impl-worker` (scoped implementation from exact
spec), `test-writer` (tests against a stated contract), `code-reviewer` (read-only;
thread-ownership §3, wire format §5, TSan/ASan hazards — run before every commit),
`explorer` (read-only search/summarize).

## Coding standards

- **C++17**, CMake ≥ 3.16. Warnings-as-errors (`-Wall -Wextra -Werror`).
- **No Abseil.** Runtime deps are exactly nghttp2, OpenSSL, protobuf-lite — found
  via find_package/pkg-config on the system; no vendoring in the default path.
- **TSan+ASan(+UBSan) clean is mandatory**, CI-enforced from M1 on. The condvar-
  heavy design makes data races the top project risk — treat sanitizer findings
  as blockers, never suppressions.
- clang-format (config at repo root) enforced in CI.
- Unit tests use **doctest** (vendored single header in `third_party/doctest/`):
  zero dependency footprint and near-zero compile cost, which fits an embedded-
  focused library better than GoogleTest. Wired into ctest.
- Python: test harness is pytest; test server uses upstream `grpcio`.
